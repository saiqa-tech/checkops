**Cons**:

- Need to run migration before using new code

**Implementation Steps**:

2. ⏳ Implement v4.0.0 models (expects UUID + SID)

5. ⏳ Run database migration (006-009)
6. ⏳ Enable and run all tests

**This is the recommended approach** - cleaner code, no technical debt.

### Option 2: Implement Compatibility Layer

**Approach**: Write code that works with BOTH v3.x and v4.0.0 schemas

**Pros**:
- Can test immediately
- Gradual migration possible
- Backward compatible

**Cons**:
- Complex code with schema detection
- Technical debt to remove later
- Harder to maintain
- More error-prone

**Not recommended** - adds unnecessary complexity.

### Option 3: Create Test Database with v4.0.0 Schema

**Approach**: Create a separate test database with v4.0.0 schema for development

**Pros**:
- Can test v4.0.0 code immediately
- No compatibility layer needed
- Clean implementation

**Cons**:
- Need to maintain two databases
- Extra setup complexity

**Good option** - allows testing while keeping code clean.

**create() Method**:
```javascript
static async create({ title, description, questions, metadata = {} }) {
  const pool = getPool();
  
  // Generate SID (human-readable)
  const counter = await getNextSIDCounter('form');
  const sid = generateSID('form', counter);
  
  // UUID is generated by database (DEFAULT gen_random_uuid())
  const result = await pool.query(
    `INSERT INTO forms (sid, title, description, questions, metadata)
     VALUES ($1, $2, $3, $4, $5)
     RETURNING *`,
    [sid, title, description, JSON.stringify(questions), JSON.stringify(metadata)]
  );
  
  return Form.fromRow(result.rows[0]);
}
```

**findById() Method**:
```javascript
static async findById(id) {
  const pool = getPool();
  
  // Resolve SID to UUID if needed
  const uuid = await resolveToUUID(id, 'forms');
  
  if (!uuid) {
    throw new NotFoundError('Form', id);
  }
  
  const result = await pool.query('SELECT * FROM forms WHERE id = $1', [uuid]);
  
  if (result.rows.length === 0) {
    throw new NotFoundError('Form', id);
  }
  
  return Form.fromRow(result.rows[0]);
}
```

### Cache Layer Changes

**Cache by both UUID and SID**:
```javascript
setForm(formData) {
  // Cache by UUID (primary)
  this.cache.set(`form:${formData.id}`, formData);
  
  // Also cache by SID for quick lookup
  this.cache.set(`form:sid:${formData.sid}`, formData);
}

getForm(id) {
  // Try UUID first
  if (isUUID(id)) {
    return this.cache.get(`form:${id}`);
  }
  
  // Try SID
  if (isSID(id, 'FORM')) {
    return this.cache.get(`form:sid:${id}`);
  }
  
  return null;
}
```

## Decision Points

### Question 1: Should we create a test database now?

**Option A**: Create test database with v4.0.0 schema now
- Pros: Can test immediately, faster development
- Cons: Extra setup, need to maintain test DB

**Option B**: Wait until models/services are implemented
- Pros: Less setup now, implement first
- Cons: Can't test database operations until later

**Recommendation**: **Option B** - implement models/services first, then create test DB

### Question 2: Should we implement all models at once or one by one?

**Option A**: Implement all models (Form, Question, Submission) at once
- Pros: Consistent approach, see full picture
- Cons: Larger change, harder to review

**Option B**: Implement one model at a time
- Pros: Easier to review, incremental progress
- Cons: Takes longer, may miss patterns

**Recommendation**: **Option B** - implement Form first, then Question, then Submission

### Question 3: Should we update services immediately after models?

**Option A**: Update models, then services, then cache
- Pros: Logical flow, easier to understand
- Cons: Can't test end-to-end until all done

**Option B**: Update model + service + cache for each entity
- Pros: Can test each entity completely
- Cons: More context switching

**Recommendation**: **Option A** - complete all models first, then services
