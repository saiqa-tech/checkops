## 🎯 Core Problem Identified

**Current Implementation is WRONG**:
- ❌ Models accept both UUID and SID (mixing concerns)
- ❌ UUID exposed in API responses (internal detail leaked)
- ❌ SID used in database queries (performance issue)
- ❌ Validation for SID (unnecessary complexity)
- ❌ Cache stores by both (wasted memory)

**This violates separation of concerns and creates confusion.**

---

## ✅ Correct Architecture

### Principle: Clear Separation of Concerns

```
┌─────────────────────────────────────────────────────────────┐
│                      USER/API LAYER                         │
│  - Users ONLY see SID (FORM-001, Q-001, SUB-001)          │
│  - API accepts SID                                          │
│  - API returns SID                                          │
│  - UUID is NEVER exposed                                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
                    [SID → UUID Resolution]
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                   INTERNAL/DATABASE LAYER                   │
│  - Database uses UUID for everything                        │
│  - All queries use UUID                                     │
│  - All joins use UUID                                       │
│  - All foreign keys use UUID                                │
│  - SID is ONLY for display                                  │
└─────────────────────────────────────────────────────────────┘
```

### Layer Responsibilities

**1. API/Service Layer** (User-Facing)
- Accept: SID only
- Return: SID only
- Resolve: SID → UUID at entry point
- Never expose UUID to users

**2. Model Layer** (Internal)
- Accept: UUID only
- Return: Objects with both UUID (internal) and SID (for display)
- Methods: `findById(uuid)`, `findBySid(sid)` (separate!)
- Database operations: UUID only

**3. Database Layer**
- Primary key: UUID
- Foreign keys: UUID
- SID: Just another column for display
- Queries: Always use UUID

## 🔍 Key Decisions

### Decision 1: toJSON() - What to Expose?

**Option A: Expose only SID** ✅ RECOMMENDED
```javascript
toJSON() {
  return {
    sid: this.sid,         // ✅ User-facing
    title: this.title,
    // ... other fields
  };
}
```

**Option B: Expose both UUID and SID** ❌ WRONG
```javascript
toJSON() {
  return {
    id: this.id,           // ❌ Internal detail leaked
    sid: this.sid,
    title: this.title,
  };
}
```

**Verdict**: Option A. UUID is internal implementation detail.

### Decision 2: Method Naming

**Option A: Separate methods** ✅ RECOMMENDED
```javascript
findById(uuid)      // Internal use
findBySid(sid)      // User-facing
updateById(uuid)    // Internal use
updateBySid(sid)    // User-facing
```

**Option B: Single method with type detection** ❌ WRONG
```javascript
find(id) {
  if (isUUID(id)) return findById(id);
  if (isSID(id)) return findBySid(id);
}
```

**Verdict**: Option A. Clear separation, no ambiguity.

### Decision 3: Foreign Keys in Responses

**Option A: Return SID for foreign keys** ✅ RECOMMENDED
```javascript
{
  sid: "SUB-001",
  formSid: "FORM-001",  // ✅ User sees form SID
  submissionData: {...}
}
```

**Option B: Return UUID for foreign keys** ❌ WRONG
```javascript
{
  sid: "SUB-001",
  formId: "550e8400-...",  // ❌ Internal UUID exposed
  submissionData: {...}
}
```

**Verdict**: Option A. Consistency - all IDs are SIDs in API.

### Decision 4: Cache Strategy

**Option A: Cache by SID only** ✅ RECOMMENDED
```javascript
cache.set(`form:${sid}`, form);
cache.get(`form:${sid}`);
```

**Option B: Cache by both UUID and SID** ❌ WRONG
```javascript
cache.set(`form:${uuid}`, form);
cache.set(`form:${sid}`, form);  // Duplicate!
```

**Verdict**: Option A. SID is the user-facing key.
